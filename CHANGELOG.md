# ATOLLSPHERE CHANGELOG
## LS AI Systems · Internal · Not for Distribution

---

## v1.2.0 — 10 April 2026

### Summary
Four architectural patches applied following analysis of Editions 013 and 014.
Triggered by: PRED-012-B PMM (CONTRADICTED), Edition 014 bypass ceiling hit (Case F),
prediction log velocity gap identified, PMM rule decay identified.

---

### Patch 1 — C3 Falsifiability Revision
**Section affected:** Section 7 — Five-Criteria Prediction Standard

**Problem:** C3 required predictions to be falsifiable but did not define how to handle silence, partial compliance, or strategic ambiguity. PRED-012-B failed because the middle ground between full adherence and full rejection was not pre-assigned to an outcome state.

**Change:** C3 now requires silence, partial compliance, and strategic ambiguity to each be pre-assigned to a named five-state outcome before the prediction window opens. If any of the three cannot be cleanly assigned, the prediction fails C3 and must be rewritten or split.

---

### Patch 2 — Bypass Override Protocol
**Section affected:** Section 5 — Format Governance

**Problem:** The bypass ceiling had no escape valve. If a new material fact arrived after the ceiling was reached, analysis was suppressed with no legitimate path forward. Case F (India Oil Waiver) in Edition 014 demonstrated this gap.

**Change:** Bypass Override Protocol added. Triggered by a new Tier 1 or Tier 2 fact that materially changes a case status tag. Ceiling resets for that case only, that edition only. Maximum one override per case per edition. If override ceiling also reached, case reduces to WATCH.

---

### Patch 3 — status_history Schema
**Section affected:** Section 15 — Prediction Log Schema (new section)

**Problem:** Prediction log recorded only final outcome state. Intermediate status changes were lost. Calibration analysis against historical hit rates was impossible without velocity data.

**Change:** status_history array added as mandatory field to every prediction log entry. Records every status change with edition reference. Entries appended only — never deleted or overwritten. All existing entries backfilled at next log update.

---

### Patch 4 — System Change Log
**Section affected:** Section 17 — System Change Log Reference (new section)

**Problem:** Rule changes generated by PMM post-mortems lived inside edition prose and decayed after one or two editions. No permanent institutional record of what the system learned from its own failures.

**Change:** SYSTEM_CHANGE_LOG.md created as permanent record. Every PMM that triggers a rule change must log an entry immediately. First entry: PRED-012-B — H1 misapplication — C3 incentive analysis requirement.
